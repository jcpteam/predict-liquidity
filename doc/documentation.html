Overview

This document outlines how to use the BTX gRPC API. Its intended audience is developers wishing to integrate with it. It provides a list of available services, their RPC methods, and the messages used in requests and responses.

You will need an understanding of gRPC, Protocol Buffers, OAuth2 and some exchange betting knowledge to work with this API effectively.

It's recommended that you use one of the recommended gRPC code generators to automatically generate the client code for your preferred programming language. This will save you a lot of time and effort.

Prerequisites

Before you can start using the BTX API, you need to be provided with three essential credentials by our onboarding team:

Required Credentials

Credential Description Usage
Account ID Your unique account identifier Must be included as the x-account-id header in ALL API requests
OAuth2 Client ID Your OAuth2 client identifier Used for OAuth2 authentication to obtain JWT tokens
OAuth2 Client Secret Your OAuth2 client secret (keep secure!) Used with Client ID for OAuth2 authentication

Getting Your Credentials

All three credentials will be provided to you by the BTX onboarding team during your account setup process. If you haven't received these yet, please contact your account manager or the onboarding team.

Security Notes

  • Never share your Client Secret - Treat it like a password
  • Store credentials securely - Use environment variables or secure credential management systems
  • Rotate credentials regularly - Contact support if you suspect your credentials have been compromised

Once you have received all three credentials from the onboarding team, you're ready to proceed to the Quick Start guide below.

Quick Start (5 minutes)

Get your first API call working in 5 minutes. This guide assumes you have your credentials.

Note: This example uses production endpoints. For initial development and testing, replace prod with staging in all URLs. See the Environments section for details.

1. Get Your JWT Token

# Replace with your actual credentials
export CLIENT_ID="your_client_id"
export CLIENT_SECRET="your_client_secret"
export ACCOUNT_ID="your_account_id"

# Get JWT token
TOKEN=$(curl -s -X POST "https://auth.prod.ex3.io/oauth2/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=$CLIENT_ID" \
  -d "client_secret=$CLIENT_SECRET" \
  -d "scope=btx/account_info" | jq -r '.access_token')

2. Make Your First API Call

# Get your account information
grpcurl \
  -H "authorization: Bearer $TOKEN" \
  -H "x-account-id: $ACCOUNT_ID" \
  -d '{}' \
  api.prod.ex3.io:443 \
  btx.api.v1.customer.account.AccountApi/GetAccount

3. Success!

If successful, you'll see your account information in JSON format. Now explore the full documentation below.

Switching Between Environments

To switch between environments, you need to:

  1. Use the appropriate OAuth2 endpoint to get your JWT token
  2. Use the corresponding gRPC API endpoint for your requests
  3. Use environment-specific credentials (your staging and production credentials are different)

Important: Your account credentials, client IDs, and client secrets are environment-specific. Staging credentials will not work in production and vice versa.

Understanding Field Types

DecimalNumber

Used for all monetary values and prices to avoid floating point precision issues.

// Example: $10.50 USD
{
  "value": 1050,
  "dps": 2
}
// Formula: actual = value / (10^dps) = 1050 / 100 = 10.50

// More examples:
// 2.00 (decimal odds): {"value": 200, "dps": 2}
// 1.95 (decimal odds): {"value": 195, "dps": 2}
// -0.5 (handicap): {"value": -50, "dps": 2}  OR  {"value": -5, "dps": 1}
// 2.5 (total goals): {"value": 250, "dps": 2}  OR  {"value": 25, "dps": 1}

Timestamps

All timestamps are Unix milliseconds (not seconds). JavaScript Date.now() compatible.

// Current time in correct format
const timestamp = Date.now(); // 1704067200000

Idempotency Keys

Used to prevent duplicate orders if network issues occur. Must be unique per request.

// Good idempotency key (UUID without dashes)
const key = crypto.randomUUID().replace(/-/g, ''); // "a1b2c3d4e5f6..."

Authentication & Authorization

A member of the onboarding team will create an account for you, you will then be provided with a few key bits of information to access BTX.

Key Description
Account ID The ID that uniquely identifies you. You must supply this as the x-account-id HTTP header with your gRPC requests.
OAuth2 Client ID The OAuth2 ID you will use to authenticate with the OAuth2 server and get a JWT token that you add to the authorization HTTP header with your gRPC requests.
OAuth2 Client Secret This is used alongside your OAuth2 Client ID to produce your JWT. This must be kept secret; we do not store this.
OAuth2 Scopes A list of 'scopes' that can be used with this client ID. Each scope will give you access to different parts of the API. Please see the scopes section below.

You can have up to 3 client IDs. Each client ID can be used to access the API with different permissions/scopes. So we can create you a client ID and secret that has just permissions to access prices. Please see the scopes section below.

OAuth2

BTX uses OAuth2 to authenticate requests. You will need to obtain a JWT token from the OAuth2 server before making any gRPC requests.

To obtain a JWT token, you will need to make a POST request to the OAuth2 server with your OAuth2 Client ID and Client Secret. The response will contain a JWT token that you can use in subsequent requests.

There are a whole host of client libraries for you to use if you want.

However, if you wish to make the request yourself, then you can use any HTTP client. The auth endpoint is https://auth.prod.ex3.io/oauth2/token


                # bash
                # please replace CLIENT_ID, CLIENT_SECRET and SCOPES with your own values.
                # SCOPES is a list separated by spaces but url encoded so %20.
                # e.g. "btx/prices%20btx/ref_data"

                curl -s -X POST "https://auth.prod.ex3.io/oauth2/token" \
                     -H "Content-Type: application/x-www-form-urlencoded" \
                     -d "grant_type=client_credentials" \
                     -d "client_id=CLIENT_ID" \
                     -d "client_secret=CLIENT_SECRET" \
                     -d "scope=SCOPES"

The response will contain a JWT token that you can use in subsequent requests. The token will be valid for the period of time in seconds in the 'expires_in' property. You must reauthenticate when this time is up.


                {
                    "access_token": "A long base64 encoded string, your access token for the API",
                    "expires_in": 43200,
                    "token_type": "Bearer"
                }

Please take the value of access_token and use it in the authorization: Bearer ACCESS_TOKEN HTTP header of your gRPC requests.

Caching & Costs

Each time you make a request to the OAuth2 server it costs us money, so please cache your JWT tokens and reuse them until they expire. You can see how long your token is valid for in the expires_in property of the response (the value is in seconds).

You should need just a few requests a day, however the first 100 are not charged, if you exceed this your account will be charged at $0.01 per request over this limit.

Scopes

The scopes allow you to control what parts of the API you can access with your OAuth2 Client ID. Each scope is a string that represents a specific area of the API. This enables you to request JWTs for different purposes. If you want to create a JWT that has limited access, you can.

Please remember that each client ID is created with a set of scopes, so you can have multiple client IDs with different scopes. This allows you to create a client ID that has access to just the prices service, or just the ref_data service. This may be useful for your own security needs.

Scope Description
btx/account_info Gives access to AccountApi-GetAccount and AccountApi-StreamTransactions
btx/active_orders Gives access to BettingApi-StreamMarketData and the ability to stream_orders
btx/ref_data Gives access to BettingApi-StreamMarketData and the ability to stream_ref_data
btx/prices Gives access to BettingApi-StreamMarketData and the ability to stream_prices
btx/place_cancel_orders Gives access to BettingApi-PlaceOrders and BettingApi-CancelOrders
btx/settled_bets Gives access to AccountApi-GetMarketSettledOrders

Market Types

BTX supports a number of market types listed below. Each one has its own nuance.

These market types are used in the StreamMarketDataRequest when streaming market data.

These types allow you to subscribe to just the market type you are interested in. The below details will help you understand the market.

Football Full Time Match Odds

Market Type: FOOTBALL_FULL_TIME_MATCH_ODDS

The market for betting on which team will win the match or if there will be a draw.

It will always have three runners, with one of the runners having a special runner ID of DRAW

The market is concluded at the end of any stoppage time. Extra-time and penalty shootouts are not included in this market.

Football Full Time Asian Handicap

Market Type: FOOTBALL_FULL_TIME_ASIAN_HANDICAP

The market for betting on which team will win with a given handicap.

When placing orders on this market you must supply the handicap for the order.

The market is concluded at the end of any stoppage time. Extra-time and penalty shootouts are not included in this market.

Football Full Time Asian Handicap Total Goals

Market Type: FOOTBALL_FULL_TIME_ASIAN_HANDICAP_TOTAL_GOALS

The market for betting on whether the total number of goals scored will be over or under a certain number.

This is a handicap market where you can bet UNDER or OVER with a handicap.

When placing orders on this market you must supply the handicap for the order.

The market is concluded at the end of any stoppage time. Extra-time and penalty shootouts are not included in this market.

Football Full Time Total Goals Over Under

Market Type: FOOTBALL_FULL_TIME_TOTAL_GOALS_OVER_UNDER

The market for betting on whether the total number of goals scored will be over or under a certain number.

The market will always have two runners: UNDER and OVER

Metadata Fields:

Key Description Example
OverUnderLine The goal line for the market "2.5", "3.0", "3.5"

The market is concluded at the end of any stoppage time. Extra-time and penalty shootouts are not included in this market.

Football Full Time Correct Score

Market Type: FOOTBALL_FULL_TIME_CORRECT_SCORE

The market for predicting the exact score of the match.

The market is concluded at the end of any stoppage time. Extra-time and penalty shootouts are not included in this market.

Cricket Match Odds

Market Type: CRICKET_MATCH_ODDS

The market for betting on who will win the cricket match. If the match is a tie, then this market is void. Super overs are not included.

This market is for fixed over games. For test matches, please see CRICKET_MATCH_ODDS_WITH_DRAW

Cricket Match Odds with Draw

Market Type: CRICKET_MATCH_ODDS_WITH_DRAW

The market for betting on who will win the cricket match where a draw is possible, like in test matches. If the match is a tie, then this market is void.

Cricket Completed Match

Market Type: CRICKET_COMPLETED_MATCH

The market for betting on whether the game will be completed or not. If there is an official result for the match, then it's decided the match has been completed. If the match has no result or is abandoned, then the market is not completed.

This market always has two runners YES and NO

Cricket Tied Match

Market Type: CRICKET_TIED_MATCH

The market for betting on whether both teams will score exactly the same number of runs.

This market always has two runners YES and NO

Cricket Innings Session Total Line

Market Type: CRICKET_INNINGS_SESSION_TOTAL_LINE

The market for betting on how many runs will be scored after a set number of overs.

The price for this market is the number of runs. If you BACK the price, you are betting that the team scores under that number of runs. If you LAY the price, you are betting the team scores over that number of runs.

Note: The metadata will contain either Session (number of overs) or Balls (number of legal balls), never both. These fields are mutually exclusive. The Balls field only counts legal deliveries - wides and no-balls are excluded from the count.

Metadata Fields:

Key Description Example
Price The actual odds price for this bet "1.95", "2.10"
Session Number of overs in the session (mutually exclusive with Balls) "6", "10", "15", "20"
Balls Number of legal balls in the session (excludes wides and no-balls; mutually exclusive with Session) "30", "60", "90", "100"
Innings Innings number "1", "2"
Team Team name for the innings (only set where possible, typically for Test matches) "India", "Australia"

Cricket Innings Total Line

Market Type: CRICKET_INNINGS_TOTAL_LINE

The market for betting on how many runs will be scored at the end of the innings.

The price for this market is the number of runs. If you BACK the price, you are betting that the team scores under that number of runs. If you LAY the price, you are betting the team scores over that number of runs.

Metadata Fields:

Key Description Example
Price The actual odds price for this bet "1.95", "2.10"
Innings Innings number "1", "2"
Team Team name for the innings (only set where possible, typically for Test matches) "India", "Australia"

The innings can be shortened due to weather etc. This market stands as long as 25 balls are bowled, the batting team is bowled out or reaches the target to win the match.

Tennis Match Odds

Market Type: TENNIS_MATCH_ODDS

The market for betting on who will win a tennis match.

Tennis Set Winner

Market Type: TENNIS_SET_WINNER

The market for betting on who will win a set in a tennis match.

Metadata Fields:

Key Description Example
Set Set number "1", "2", "3", "4", "5"

The set number is also included in the market name for clarity.
